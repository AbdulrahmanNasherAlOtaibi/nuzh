# استراتيجية البناء الآمن لمنصة نُزه

> خطة تنفيذية مرحلية. كل بند قابل للتحقق (✅). نُنفّذها معاً على مراحل — لا يُطلق التطبيق للسوق قبل اكتمال المرحلتين 1 و2 على الأقل.

الحالة الحالية باختصار: الأساس سليم (تجزئة scrypt للكلمات، كوكيز HttpOnly + SameSite=Lax، استعلامات مُعدّة تمنع حقن SQL، فصل صلاحيات user/provider/admin، رقابة على رفع الصور). النواقص الإنتاجية موزّعة على المراحل أدناه.

---

## المرحلة 0 — الأساسات والحوكمة (قبل أي كود جديد)

- [ ] **إدارة الأسرار**: نقل كل القيم الحساسة إلى `.env` (غير مُتتبَّع) — `SESSION_SECRET`, مفاتيح بوابة الدفع, مفاتيح توثيق الهوية. توليد `SESSION_SECRET` عشوائي 256-بت.
- [ ] **فرض تغيير كلمة مرور الأدمن** عند أول دخول (بدل `admin123` الثابتة)، ومنع تشغيل الإنتاج بكلمة المرور الافتراضية.
- [ ] **نموذج التهديدات (Threat Model)**: توثيق الأصول (بيانات المستخدمين، الأموال، بيانات الشركات)، والمهاجمين المحتملين، وأسطح الهجوم (الحجز، الدفع، رفع الصور، لوحة الأدمن).
- [ ] **تصنيف البيانات**: تحديد ما هو شخصي (PII) وما هو مالي، وسياسة الاحتفاظ والحذف.
- [ ] **مراجعة الاعتماديات**: تفعيل `npm audit` + Dependabot، وتثبيت إصدارات محددة.

## المرحلة 1 — تحصين التطبيق (حرجة — قبل الإطلاق)

### المصادقة والجلسات
- [ ] إضافة علم **`Secure`** للكوكي في الإنتاج + `__Host-` prefix.
- [ ] **انتهاء صلاحية الجلسات** الفعلي (تدوير التوكن، حد أقصى للعمر، إبطال عند تغيير كلمة المرور).
- [ ] **حماية CSRF**: توكن مزدوج (double-submit) أو رفع SameSite إلى `Strict` للعمليات الحسّاسة + التحقق من `Origin`.
- [ ] **تحديد معدّل المحاولات** على `/login` و`/register` (قفل تصاعدي بعد N محاولات) — الجدول `logins` موجود، نبني عليه.
- [ ] **تحقق البريد فعلياً** (حالياً `email_verified=1` تلقائي) — رمز OTP أو رابط تفعيل.
- [ ] سياسة كلمات مرور أقوى + كشف الكلمات المسرّبة (haveibeenpwned k-anonymity اختياري).

### تحصين الـ API
- [ ] **رؤوس الأمان** عبر `helmet`: CSP، HSTS، X-Frame-Options، X-Content-Type-Options، Referrer-Policy.
- [ ] **تحديد معدّل عام** (`express-rate-limit`) لكل الـ endpoints + تشديد على الكتابة.
- [ ] **تحقق مدخلات صارم** بمخطط (`zod`) على كل جسم طلب — بدل الفحص اليدوي المتناثر.
- [ ] تقليل حد جسم JSON من **12mb** إلى ~256kb، ورفع الصور عبر مسار منفصل بحدود صارمة (نوع/حجم/أبعاد + فحص السحر magic-bytes بدل الثقة بـ `data:image`).
- [ ] **ضبط CORS** صراحةً (أصل واحد موثوق) بدل الاعتماد على same-origin ضمناً.
- [ ] فرض **HTTPS** وإعادة التوجيه، و`trust proxy` الصحيح خلف الموازِن.
- [ ] مراجعة **التحكم بالصلاحيات على مستوى الكائن** (IDOR): التأكد أن كل مسار `/my/*` و`/provider/*` يتحقق من الملكية (مطبّق جزئياً — نراجعه بالكامل).

### الحماية من الثغرات الشائعة (OWASP Top 10)
- [ ] XSS: التطبيق React (هروب تلقائي) — مراجعة أي `dangerouslySetInnerHTML` ومحتوى المستخدم في الإعلانات/المقالات.
- [ ] فحص شامل لـ **حقن SQL** (مؤمّن حالياً بالاستعلامات المُعدّة — نثبّت القاعدة: ممنوع أي تركيب نصّي لاستعلام).
- [ ] منع **رفع الملفات الخبيثة** وتخزينها خارج قاعدة البيانات (حالياً base64 داخل DB — ننقلها لتخزين كائني/مجلد آمن).

## المرحلة 2 — الدفع والامتثال المالي (حرجة — قبل استقبال أموال حقيقية)

- [ ] **بوابة دفع معتمدة** (Moyasar / Tap / HyperPay / PayTabs) — التطبيق حالياً واجهة فقط بلا دفع حقيقي.
- [ ] **عدم تخزين بيانات البطاقات إطلاقاً** (PCI-DSS SAQ-A): الدفع عبر iframe/hosted fields من البوابة.
- [ ] **التحقق من الدفع من الخادم** عبر webhook موقّع من البوابة (لا نثق بنتيجة العميل).
- [ ] **حماية idempotency** لمنع الحجز/الخصم المزدوج.
- [ ] سياسة **الاسترداد والإلغاء** آلياً عبر البوابة (مطبّق منطقياً — نربطه بالبوابة).
- [ ] إن تطلّب النموذج حفظ أموال وسيطة (Escrow): مراجعة متطلبات **ساما SAMA** والترخيص.

## المرحلة 3 — توثيق الهوية والتكامل الحكومي

- [ ] **نفاذ الوطني الموحّد (NIC)** لتوثيق هوية المستخدمين قبل العمليات المالية.
- [ ] **تصاريح المحميات**: التكامل مع المركز الوطني لتنمية الحياة الفطرية (حالياً محاكاة برقم QR).
- [ ] التحقق من **السجل التجاري/الترخيص السياحي** للشركات المزوّدة قبل التفعيل.

## المرحلة 4 — حماية البيانات والامتثال (PDPL)

- [ ] الامتثال لـ **نظام حماية البيانات الشخصية (PDPL)** السعودي: سياسة خصوصية، أساس المعالجة، حقوق صاحب البيانات (وصول/حذف/تصحيح).
- [ ] **تشفير البيانات أثناء الراحة** (تشفير القرص/قاعدة البيانات) وأثناء النقل (TLS 1.2+).
- [ ] **تقليل البيانات**: عدم جمع أكثر من اللازم، وإخفاء/تعمية الحقول الحسّاسة في السجلات.
- [ ] **سياسة الاحتفاظ والحذف** وحذف الحساب الفعلي (Right to erasure).
- [ ] **استضافة البيانات داخل المملكة** حسب متطلبات التنظيم.
- [ ] مراجعة **ضوابط الأمن السيبراني (NCA ECC)** إن انطبقت.

## المرحلة 5 — البنية التحتية والنشر الآمن

- [ ] **ترقية قاعدة البيانات** من SQLite (ملف) إلى **PostgreSQL مُدار** مع نسخ احتياطي آلي واستعادة مُختبرة.
- [ ] **أسرار في مدير أسرار** (Doppler/Vault/متغيرات المنصة) لا في الكود.
- [ ] **مبدأ الأقل امتيازاً** لمستخدم قاعدة البيانات والخدمات.
- [ ] **جدار حماية تطبيقات ويب (WAF)** + حماية DDoS (Cloudflare مثلاً).
- [ ] **حاوية Docker مقواة**: مستخدم غير جذر، صورة صغيرة، فحص الصور (Trivy).
- [ ] **بيئات منفصلة** (تطوير/تجربة/إنتاج) بأسرار مختلفة.
- [ ] **CI/CD آمن**: فحص أسرار مُسرّبة، SAST، `npm audit` بوابة قبل الدمج.

## المرحلة 6 — المراقبة والاستجابة

- [ ] **سجلات مركزية** للأحداث الأمنية (دخول، تغيير صلاحيات، عمليات مالية) — الجدول `activity_log` موجود، نوسّعه.
- [ ] **تنبيهات** على الأنشطة المريبة (محاولات دخول فاشلة متكررة — مطبّق في لوحة الأدمن، نربطه بتنبيه فعلي).
- [ ] **خطة استجابة للحوادث** (من يفعل ماذا عند اختراق/تسريب) + جهة تواصل.
- [ ] **مراقبة توفّر** (uptime) وأداء وأخطاء (Sentry).

## المرحلة 7 — الاختبار والتحقق المستمر

- [ ] **اختبارات آلية** للمنطق الحسّاس (الصلاحيات، الملكية، الدفع، آلة حالات الحجز).
- [ ] **SAST** (تحليل ثابت — CodeQL/Semgrep) في CI.
- [ ] **DAST/فحص ثغرات** دوري (OWASP ZAP).
- [ ] **اختبار اختراق** خارجي قبل الإطلاق وبعد كل تغيير كبير.
- [ ] **مراجعة كود أمنية** لكل PR يمسّ المصادقة/الدفع/الصلاحيات.

## أولوية التنفيذ المقترحة
1. **المرحلة 1** (تحصين التطبيق) — نبدأ بها فوراً، أغلبها كود نكتبه معاً بسرعة.
2. **المرحلة 2** (الدفع) — بالتوازي، تحتاج قرارك في اختيار البوابة.
3. **المرحلة 4 + 5** (الامتثال والبنية) — قبل الإطلاق الفعلي.
4. **المرحلة 3** (نفاذ/التكامل الحكومي) — حسب توفّر الوصول للـ APIs الرسمية.
5. **المرحلة 6 + 7** — مستمرة بعد الإطلاق.

## قرارات أحتاجها منك للبدء
- بوابة الدفع المفضّلة (Moyasar / Tap / HyperPay / PayTabs / أخرى).
- جهة الاستضافة (DigitalOcean / AWS بمنطقة البحرين / stc cloud / أخرى).
- هل توثيق الهوية عبر نفاذ متاح لك الآن أم نؤجّله بعلامة «قريباً»؟
- نموذج الأموال: هل تحتفظ المنصة بالمبلغ (Escrow) أم تحويل مباشر للمزوّد؟ (يحدد متطلبات ساما).
